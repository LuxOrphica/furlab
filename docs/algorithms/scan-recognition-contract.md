# Контракт на доработку распознавания сканов FurLab (line / contour / ID)

## 1. Цель
Стабильно определять на скане:
- идентификатор куска (`FL-SCR-xxxxxx`),
- контур куска,
- линию маркера на мездре и угол направления ворса.

Ключевой приоритет: **линия маркера и направление ворса**.

## 2. Архитектура детектора линии (обязательная)
Pipeline:

`source raster -> ROI мездры -> карта тёмных штрихов -> connected components -> фильтр по признакам -> best candidate -> angle + confidence`

Требования:
1. Вход только из исходного raster (не из UI-оверлеев, не из zoom/viewport).
2. Поиск только внутри мездры, с исключением края и зоны QR.
3. Тёмный штрих ищется как **локальная аномалия темноты**, а не только глобальным порогом яркости.
4. После бинаризации — компоненты и фильтрация по признакам.
5. Возвращать `angle` и `confidence`.
6. Если уверенность низкая — статус `unknown`, а не случайный угол.

## 3. ROI и исключающие зоны
- `pieceMask` обязателен.
- `innerMask` формируется аккуратной эрозией от края мездры.
- Из поиска исключаются:
  - зона QR,
  - подпись под QR,
  - край с ворсом.

Рекомендуемые стартовые параметры:
- `edgeMarginPx = max(12, round(min(w,h) * 0.02))`
- `qrMarginPx = 28..40`

## 4. Dark-map (локальная темнота)
Рекомендуемая схема:
1. `L = grayscale(image)`
2. подавление мелкой текстуры (bilateral / guided / мягкий blur)
3. `Bg = локальный фон` (крупный blur или morph-open)
4. `Dark = Bg - L`
5. Порог по ROI робастно (`median + k * MAD`), а не глобальный fixed threshold.

## 5. Признаки кандидата линии
Минимальный набор:
- `length`
- `aspect` (вытянутость)
- `straightness`
- `meanWidth`
- `meanDark`, `p90Dark`
- `distToEdge`
- `distToQR`
- `solidity` / устойчивость формы

Отбраковка обязательна для:
- тонких волосков,
- рваных кусков у края,
- элементов QR/текста,
- слишком коротких/широких компонентов.

## 6. Выход и статусы
Обязательные поля:
- `lineFound: boolean`
- `lineSource: string`
- `lineAngleDeg: number | null`
- `lineConfidence: number | null`
- `napDirectionStatus: "found" | "unknown" | "not_found"`

Правило:
- найдено уверенно -> `found`
- есть кандидат, но низкая уверенность -> `unknown`
- кандидата нет -> `not_found`

## 7. Диагностика (обязательная)
Для каждого кадра сохранять/показывать этапы:
1. `01_source`
2. `02_search_mask` (ROI)
3. `03_dark_map`
4. `04_binary_candidates`
5. `05_components_overlay`
6. `06_best_candidate`

В тех-выводе минимум:
- `mode/stage`
- пороги
- число кандидатов
- причины отбраковки
- `confidence`
- времена этапов (`decode`, `mask`, `auto`, `qr`, `total`)

## 8. Производительность
Целевые рамки (сканы 2-4 MP):
- `auto <= 1200 ms` (желательно)
- `total <= 3000 ms` (рабочая цель)

Если budget превышен — это фиксируется в telemetry, но не должно скрывать причину отказа детектора.

## 9. Идентификатор (QR/OCR)
- Основной источник ID: OCR текста формата `FL-SCR-xxxxxx`.
- QR — дополнительный канал.
- Если QR не считан, но OCR стабилен, процесс не блокируется.

## 10. Критерии приёмки
Решение принимается, если одновременно:
1. На очевидных кадрах с чёрной линией детектор выдаёт `found` и адекватный угол.
2. На сомнительных кадрах выдаёт `unknown/not_found`, без ложной линии.
3. Маска/контур не захватывают явный фон и не ломают ROI линии.
4. В тех-информации есть прозрачная диагностика по всем этапам.
5. Нет деградации UI (фризы, неуправляемая задержка).

## 11. Запрещённые упрощения
Нельзя:
- искать линию по UI-слоям,
- опираться только на один global threshold,
- возвращать угол при отсутствии уверенного кандидата,
- считать задачу закрытой без этапных артефактов.
