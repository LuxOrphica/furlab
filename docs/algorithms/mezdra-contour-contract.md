# Контракт, ориентированный на результат: внешний контур мездры

## 1. Цель

Цель этапа — получить на реальных сканах корректный внешний контур именно мездры, без захвата меховой бахромы.

Метод реализации не является предметом приемки.  
Исполнитель вправе использовать любой подход, если он:
- дает требуемый результат;
- не вызывает фризов UI;
- детерминирован;
- не дает недопустимой регрессии на legacy-наборе.

---

## 2. Что считается результатом

Результатом этапа считается pipeline, который на вход принимает изображение лоскута, а на выходе возвращает:

- `finalMask` мездры;
- внешний контур мездры;
- `bbox`;
- `area`;
- `processingTimeMs`;
- флаги `fallbackUsed` и `timeoutHit`.

Главный объект выделения:
- **мездра**,
- а не общий силуэт лоскута,
- не меховая бахрома,
- не просто “объект на фоне”.

---

## 3. Что не важно для приемки

Для приемки не имеет значения:
- rule-based это или hybrid;
- сколько внутренних стадий у pipeline;
- какие именно эвристики, guards или refine используются.

Это допустимо менять без отдельного согласования, если соблюдены критерии результата.

---

## 4. Главный критерий приемки

Главный критерий:
- внешний контур должен идти по границе мездры,
- а не по меховой бахроме,
- и это должно подтверждаться на фиксированном наборе реальных сканов.


---

## 5. Обязательные ограничения

Кроме главного критерия, обязательно должны выполняться условия:

- без фризов UI;
- детерминированный результат на одинаковом входе;
- fallback при ошибке или таймауте;
- отсутствие недопустимой регрессии на legacy-сканах.

---

## 6. Приемочный набор

Для приемки используется фиксированный набор:
- 4 legacy-скана;
- 8 реальных сканов;
- 4 сложных реальных скана.

Итого: 16 кейсов.

Желательно наличие golden set с ручной разметкой, но финальная оценка в первую очередь идет по главному визуально-технологическому критерию:
корректность внешнего контура мездры.

---

## 7. Приемочные условия

Решение принимается, если одновременно выполнено следующее:

### 7.1. Качество контура
На реальных сканах контур мездры:
- визуально соответствует границе мездры;
- не захватывает выраженную меховую бахрому;
- не срывается в узкий или разрушенный контур;
- не “съедает” значимые участки мездры.

### 7.2. Стабильность
- одинаковый вход дает одинаковый результат;
- нет случайных колебаний маски между запусками.

### 7.3. Производительность
Для изображений 2–4 MP:
- `p95(processingTimeMs) <= 250 ms`
- `max(processingTimeMs) <= 600 ms`

### 7.4. Safety
- при ошибке или таймауте возвращается безопасный fallback-результат;
- UI не блокируется.

### 7.5. Legacy
На legacy-наборе не допускается заметная регрессия:
- `abs(deltaAreaPct) <= 3%`
- `abs(deltaBBoxWPct) <= 2%`
- `abs(deltaBBoxHPct) <= 2%`

---

## 8. Что считается неприемлемым

Решение не принимается, если наблюдается хотя бы один из случаев:

- контур идет по меху, а не по мездре;
- на части реальных кейсов контур срывается или схлопывается;
- улучшение достигается только через постоянный fallback;
- UI фризит;
- результат недетерминирован;
- есть регрессия на legacy-наборе выше допусков.

---

## 9. Обязательные автотесты

Исполнитель обязан предоставить автотесты минимум на следующее:

### 9.1. Smoke tests
- pipeline возвращает валидный результат;
- `finalMask` не пустая;
- `bbox` валиден;
- `area > 0`.

### 9.2. Determinism tests
- одинаковый вход + одинаковая конфигурация => одинаковый результат.

### 9.3. Timeout / fallback tests
- при искусственном timeout срабатывает fallback;
- pipeline не падает;
- выставляются корректные флаги.

### 9.4. Legacy regression tests
- отклонения на legacy-наборе не выходят за согласованные допуски.

### 9.5. Performance tests
- `p95` и `max` укладываются в лимиты на test set.

---

## 10. Что должен сдать исполнитель

Исполнитель должен передать:

- рабочую реализацию pipeline;
- набор автотестов;
- результаты прогона на фиксированном наборе;
- overlay “исходник + contour”;
- краткое описание выбранного подхода;
- перечень известных ограничений.

---

## 11. Право выбора метода

Исполнитель не ограничен текущими ветками V2/V3 как единственным способом достижения результата.

Допускается:
- доработка существующего pipeline;
- замена внутренней логики сегментации;
- переход к material-aware segmentation;
- hybrid-подход;
- иная реализация, если она удовлетворяет условиям приемки.

---

## 12. Итоговый принцип

Принимается не “архитектурно красивая версия” и не “правильно названный pipeline”.

Принимается только решение, которое дает:
- корректный внешний контур именно мездры,
- устойчиво,
- без фризов,
- без недопустимой регрессии.
