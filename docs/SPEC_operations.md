## 1. Создание зоны

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Открыт проект с хотя бы одной деталью (`Part`).
- Пользователь находится на экране детали.

### Что происходит

**Вариант А — Рисование:**  
Пользователь рисует замкнутый контур на канвасе детали. Инструмент рисования собирает точки и замыкает полигон. Создаётся запись `Zone` с `zoneContour` = нарисованный полигон, `pileDirectionMode` = `AlongGrain` по умолчанию.

**Вариант Б — Импорт DXF:**  
Пользователь выбирает DXF-файл. Парсер читает геометрию из файла. Каждый замкнутый контур импортируется как отдельная `Zone` на детали. Система не фильтрует по слоям DXF (все замкнутые контуры принимаются как зоны). Поля `materialId`, `pileDirectionMode` остаются пустыми до ручного заполнения.

### Выходное состояние
- В БД/проекте создана запись `Zone` с заполненным `zoneContour`.
- `ZoneState` = `NoResult`.
- `materialId` = NULL (требует назначения).

### Крайние случаи
- Незамкнутый контур при рисовании — операция блокируется, пользователю показывается предупреждение.
- DXF с несколькими вложенными контурами — создаётся несколько зон.
- Контур зоны выходит за границы детали — система принимает его без усечения (усечение выполняется при генерации выкладки).
- Дублирующий импорт того же DXF — создаёт новые зоны поверх существующих (дубликаты не детектируются автоматически).

## 2. Разбивка зоны интарсией (applyIntarsiaFragmentsToZone)

**Статус:** ✅ (конвертация в зоны — 📋 Planned по PRD L-13)  
**Модуль:** FurLab Studio

### Входное состояние
- Зона существует (`Zone.id` известен).
- Загружены SVG-контуры интарсии (отдельная операция, см. п.6).
- Зона имеет хотя бы один активный результат выкладки или интарсия применяется напрямую к зоне.

### Что происходит
1. SVG-контуры интарсии приводятся в координатное пространство зоны (трансформация scale + translate).
2. Каждый контур интарсии клиппируется по `Zone.zoneContour` — операция `intersect`.
3. Остаточная область зоны (вне контуров интарсии) вычисляется как `diffMulti` — операция геометрического вычитания.
4. Формируются фрагменты (`Fragment`) для каждой пересекающейся области.
5. Фрагменты сохраняются с `zoneId` родительской зоны.

**Конвертация в зоны (Planned):** каждый фрагмент интарсии может быть преобразован в отдельную `Zone` со своим `materialId` и направлением ворса. Механизм `intersect/diffMulti` запланирован, но логика создания дочерних зон не завершена.

### Выходное состояние
- Набор `Fragment` с контурами `fragmentContour`, производными от SVG-контуров интарсии.
- `ZoneState` → `HasResult`.

### Крайние случаи
- SVG-контур полностью выходит за пределы зоны — фрагмент с нулевой площадью, исключается нормализацией.
- Пересекающиеся SVG-контуры между собой — обрабатываются в порядке z-index SVG; нижний контур усекается верхним.
- Зона без назначенного материала — интарсия применяется, но экспорт будет неполным.

## 3. Удаление зоны

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона существует.
- Зона может иметь привязанные `Layout`, `LayoutRun`, `Fragment`.

### Что происходит
1. Пользователь выбирает зону в дереве и инициирует удаление.
2. Каскадное удаление: `Fragment` → `LayoutRun` → `Layout` → `Zone`.
3. Если к фрагментам зоны привязаны `LayoutRunScrapPlacement` — связанные `ScrapReservation` снимаются (статус кусков возвращается в `Available`), `ScrapTransaction` с типом `Release` записывается.

### Выходное состояние
- Запись `Zone` и все связанные записи удалены.
- Зарезервированные куски переведены в `Available`.

### Крайние случаи
- Кусок уже в статусе `Used` (подтверждено использование) — удаление зоны не меняет статус куска, создаётся транзакция `Release` с пометкой о ревизии.
- Удаление детали (`Part`) — каскадно удаляет все зоны детали с теми же правилами.

## 4. Назначение материала зоне

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона существует.
- В библиотеке есть хотя бы один `FurMaterial`.

### Что происходит
1. Пользователь выбирает материал из библиотеки и назначает его зоне.
2. `Zone.materialId` обновляется на выбранный `FurMaterial.id`.
3. Визуализация зоны обновляется — паттерн меха отображается с параметрами выбранного материала (`pileLengthMm`, `pileDensityPerIn2`, `color` и т.д.).
4. Если у зоны уже есть результат выкладки (`ZoneState = HasResult`) и материал изменён — `ZoneState` переводится в `NeedsRegeneration`.

### Выходное состояние
- `Zone.materialId` = ID выбранного материала.
- `ZoneState` = `NeedsRegeneration` (если был результат) или остаётся `NoResult`.

### Крайние случаи
- Назначение материала, отсутствующего в БД инвентаря — разрешено; инвентарная выкладка из этого материала будет возвращать пустой пул кандидатов.
- Материал с `NULL` размерами (`blankWidthMaxMm`) — не блокирует назначение, но может дать некорректный предпросмотр.

## 5. Регулярная выкладка (RegularLayout)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона с заданным `zoneContour` и `materialId`.
- Конфигурация `Layout` с `layoutType = RegularLayout`.
- `patternId` выбран из библиотеки паттернов. Поддерживаемые паттерны: продольный (`longitudinal`), ёлочка (`herringbone`), смещение рядов (`offset`), радиальный (`radial`).
- `patternParams` заданы: шаг (мм), угол (°), для радиальной — центр и шаг колец.
- Опционально: `normalizeRules` (`minFragmentWidthMm`, `seamAllowanceReserveMm` и др.).

### Что происходит
1. Создаётся `LayoutRun`, фиксируется `paramsSnapshot` — полная копия `Layout.params` на момент запуска.
2. Генерируется структура паттерна в пространстве зоны согласно `patternParams`.
3. Результат клиппируется по `Zone.zoneContour`.
4. Применяются `normalizeRules`:
   - удаляются фрагменты уже `minFragmentWidthMm` или `minFragmentLengthMm`;
   - при `mergeSmallFragments=true` мелкие соседние фрагменты объединяются;
   - контуры упрощаются с допуском `simplifyToleranceMm`.
5. Для каждого фрагмента создаётся запись `Fragment` с `fragmentContour` и `areaMm2`.

**Особенности паттернов:**
- **Продольная/поперечная:** прямоугольная сетка с шагом и углом поворота.
- **Ёлочка (`herringbone`):** зигзагообразные полосы, угол задаётся относительно направления ворса зоны.
- **Смещение рядов (`offset`):** каждый чётный ряд смещён на половину шага.
- **Радиальная:** концентрические кольца и радиальные секторы; центр задаётся вручную или вычисляется как центроид зоны.

### Выходное состояние
- Набор `Fragment` в БД проекта.
- `ZoneState` = `HasResult`.
- `LayoutRun.resultSnapshot` содержит метрики: количество фрагментов, суммарная площадь, коэффициент заполнения.

### Крайние случаи
- Зона слишком мала для одного шага паттерна — создаётся один фрагмент, совпадающий с зоной.
- Угол ворса зоны не задан при `pileDirectionMode = Custom` — выкладка разрешена, но контроль ворса не выполняется.
- `normalizeRules` удаляет все фрагменты (зона слишком узкая) — `LayoutRun` сохраняется с пустым результатом, `ZoneState = HasResult` (пустой), пользователю выводится предупреждение.

## 6. Интарсия — загрузка SVG, применение к зоне (IrregularLayout)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона с заданным `zoneContour`.
- SVG-файл с замкнутыми путями (`<path>` или `<polygon>`).

### Что происходит
1. SVG-файл загружается, из него извлекаются замкнутые контуры.
2. Контуры масштабируются/трансформируются в систему координат зоны (пространство мм, ось Y вниз).
3. Создаётся `Layout` с `layoutType = IrregularLayout`, `sourceType = import`, `contours = []` (нормализованные контуры).
4. При необходимости контуры упрощаются с `simplifyToleranceMm`.
5. Запуск `LayoutRun` → клиппинг каждого контура по `Zone.zoneContour` → формирование `Fragment`.
6. `normalizeRules` применяются аналогично регулярной выкладке.

**Умное сопоставление (L-11):** система автоматически масштабирует SVG-контуры под bounding box зоны, сохраняя пропорции. Пользователь может скорректировать масштаб и позицию вручную.

### Выходное состояние
- `Fragment` с контурами, производными от SVG-форм, усечёнными по границе зоны.
- `ZoneState` = `HasResult`.

### Крайние случаи
- SVG содержит незамкнутые пути — они игнорируются без ошибки.
- SVG с единицами в пикселях вместо мм — требует явного указания `dpi` или масштабного коэффициента; при отсутствии принимается `dpi=96`.
- Контуры полностью вне зоны — такие фрагменты имеют нулевую площадь и исключаются нормализацией.

## 7. Инвентарная выкладка ручная (InventoryLayout, manualAssist)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона с `materialId`.
- В инвентаре есть `ScrapPiece` со статусом `Available` и совпадающим `materialId`.
- `Layout` с `layoutType = InventoryLayout`, `placementStrategy = manualAssist`.

### Что происходит
1. Пользователь открывает список кандидатов (фильтрация по материалу, качеству, размеру).
2. Для каждого куска отображается контур с визуализацией на канвасе зоны.
3. Пользователь перетаскивает кусок, поворачивает (задаёт `rotationDeg`), позиционирует (`offsetXmm`, `offsetYmm`).
4. При подтверждении размещения: контур куска клиппируется по `Zone.zoneContour`, создаётся `Fragment`, создаётся запись `LayoutRunScrapPlacement`.
5. Повторяется для следующего куска до заполнения зоны или решения пользователя остановить.
6. `LayoutRun` фиксируется с `paramsSnapshot`.

### Выходное состояние
- `Fragment` для каждого размещённого куска.
- `LayoutRunScrapPlacement` записи с трансформацией (`rotationDeg`, `offsetXmm`, `offsetYmm`, `resultContourSnapshot`).
- Статус кусков остаётся `Available` до явного сохранения проекта (резервирование — при сохранении, см. п. 13).

### Крайние случаи
- Пользователь размещает кусок вне зоны — кусок принимается, но пересечение с зоной = 0, фрагмент не создаётся.
- Перекрытие двух кусков — система разрешает перекрытие (визуальное предупреждение); фрагменты рассчитываются без автоматического вычитания.
- Кусок в статусе `Reserved` другим проектом — отображается с предупреждением, но не блокируется для ручного выбора.

## 8. Инвентарная выкладка NFP+SA (InventoryLayout, nfp_sa)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона с `zoneContour`, `materialId`, `napDirectionDeg`.
- `InventoryLayoutConfig` с параметрами:
  - `maxCandidates` (рекоменд. 300),
  - `filters`: `materialId`, `napTargetDeg` ± `napToleranceDeg` (дефолт ±15°), `scrapStatus = Available`, `scrapQuality`,
  - `constraints`: `seamAllowanceReserveMm` (дефолт 3 мм), `maxSolveMs` (дефолт 90 000 мс), `seed` (опц.).

### Что происходит
1. Формируется пул кандидатов: запрос к БД инвентаря по `filters`, не более `maxCandidates` кусков.
2. **Растеризация:** зона и каждый кусок-кандидат растеризуются в бинарные маски с шагом ячейки 3 мм. Вычисляется `zoneMask` и `zoneCells`.
3. **Greedy Warm Start:** кандидаты сортируются по убыванию площади. Для каждого — 16 попыток случайного размещения; выбирается позиция с максимальным покрытием незанятой области. Формируется начальное решение `R₀`.
4. **Simulated Annealing (SA):** оптимизация `R₀` операциями TRANSLATE, ROTATE, SWAP, ADD с критерием Метрополиса. Функция энергии: `E(R) = −covCells(R) + λ·overlapCells(R) − μ·|R|`. Расписание температур: T₀=2.0, α=0.9995, T_min=0.01. Остановка при T < T_min или истечении `maxSolveMs`.
5. **Постобработка:** для каждого куска из решения вычисляется ядро `core_i = T_i(inset(p_i, seamAllowanceReserveMm)) ∩ Z`, затем вычитаются уже занятые ядра. Результирующая геометрия — `frag_i`. Если `frag_i` = ∅ — кусок исключается. Если `frag_i` состоит из нескольких связных компонент — каждая становится отдельным `Fragment`.
6. Вычисляется `coveredRatio = 1 − Area(holes)/Area(Z)`.
7. **До Apply:** результат показывается в режиме предпросмотра. В БД ничего не фиксируется.
8. **Apply:** пользователь подтверждает — создаются `Fragment`, `LayoutRun` (с `paramsSnapshot`), `LayoutRunScrapPlacement`. Статусы кусков не меняются до сохранения проекта.

### Выходное состояние
- Набор `Fragment` с трассируемостью к `ScrapPiece`.
- `LayoutRun.paramsSnapshot` содержит полный снимок входных параметров (воспроизводимость).
- `LayoutRunScrapPlacement` для каждой пары `(fragmentId, scrapPieceId)`.
- `coveredRatio` в `resultSnapshot`.

### Крайние случаи
- Пул кандидатов пуст (нет подходящих по фильтрам) — солвер возвращает пустое решение, пользователь получает предупреждение.
- `maxSolveMs` истёк до достижения `coveredRatio` — возвращается лучшее найденное решение на момент остановки.
- Все ядра кусков меньше минимального порога утилизации (`util_min = 0.15`) — такие куски исключаются из результата.
- Кусок с `scrapContour = NULL` — исключается из пула на этапе фильтрации.
- SA-солвер работает в отдельном потоке; прогресс транслируется через SSE-поток (не блокирует UI).

## 9. Экспорт лекал в CLO 3D (DXF/GLTF)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Зона в состоянии `ZoneState = HasResult` (есть фрагменты).
- Опционально: `FurMaterial` с заполненными параметрами визуализации для GLTF.

### Что происходит

**Экспорт DXF (R-05):**
1. Для каждого `Fragment` из выбранных зон берётся `fragmentContour`.
2. Контуры экспортируются в файл DXF (формат поддерживается CLO 3D для импорта как pattern pieces).
3. Каждый фрагмент — отдельная замкнутая полилиния в DXF.
4. Имена слоёв: `[partName]_[zoneName]_[fragmentIndex]`.

**Экспорт GLTF (R-06):**
1. Из `FurMaterial.properties` формируется материальная модель: цвет, длина ворса, направление, блеск.
2. Генерируется GLTF-файл с текстурой/шейдером меха для CLO 3D.
3. GLTF предназначен для визуализации поверхности изделия в CLO 3D, не для раскроя.

### Выходное состояние
- Файл `.dxf` для импорта в CLO 3D как pattern pieces.
- Файл `.gltf` для визуализации материала.
- `ZoneState` → `Exported`.

### Крайние случаи
- Фрагменты с площадью 0 — исключаются из экспорта без ошибки.
- Зона без материала — DXF экспортируется, GLTF не генерируется (предупреждение).
- Интеграция с CLO 3D — только файловый обмен; прямого API нет. Пользователь импортирует DXF вручную в CLO 3D.
- Координаты: все контуры в миллиметрах; CLO 3D принимает мм без перевода.

## 10. Спецификация (отчёт)

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Проект с хотя бы одной зоной в состоянии `HasResult`.

### Что происходит
1. Агрегируются все `Fragment` по иерархии `Part → Zone → Fragment`.
2. Для каждого фрагмента берутся: `areaMm2`, `fragmentContour` (для bbox), `zoneId` → `materialId`.
3. Формируется сущность `Specification` со списком `SpecItem`:
   - деталь / зона / номер фрагмента / материал / площадь / габариты.
4. Если к фрагменту привязан `LayoutRunScrapPlacement` — добавляется инвентарный тег (`inventoryTag`) куска.
5. Отчёт доступен для отображения в UI и экспорта (CSV/печать).

**Ведомость инвентарных кусков (R-04):** отдельный разрез — список `ScrapPiece` с их `inventoryTag`, `storageLocationId`, количеством фрагментов, суммарной площадью использования.

### Выходное состояние
- `Specification` объект, доступный в UI.
- Экспортный файл (CSV или печатная форма).

### Крайние случаи
- Зоны без результата — исключаются из спецификации (или отображаются с пустой строкой).
- Один кусок используется в нескольких фрагментах (NFP+SA с несколькими компонентами) — в ведомости отображается одна строка с суммарной площадью.

## 11. Сохранение/загрузка проекта

**Статус:** ✅  
**Модуль:** FurLab Studio

### Входное состояние
- Открыт проект (новый или загруженный ранее).

### Что происходит

**Сохранение:**
1. Все сущности проекта (`Part`, `Zone`, `Layout`, `LayoutRun`, `Fragment`) сериализуются в JSON-файл проекта.
2. Резервная копия текущего файла сохраняется с расширением `.backup`.
3. **Резервирование кусков (см. п.13):** для каждого `LayoutRunScrapPlacement` проверяется: нет ли уже активного `ScrapReservation` за этим куском в этом проекте. Если нет — создаётся `ScrapReservation` в БД инвентаря, статус `ScrapPiece` → `Reserved`, транзакция `Reserve` записывается.
4. При недоступности Access БД — резервирование логируется в JSON-fallback файл (`reservations.json`).

**Загрузка:**
1. JSON-файл проекта десериализуется.
2. Производится сверка `LayoutRunScrapPlacement` с БД: если кусок имеет статус `Used` или `Discarded` — пользователь получает предупреждение о несоответствии.
3. `ZoneState` пересчитывается по наличию `LayoutRun` и совпадению `paramsSnapshot`.

### Выходное состояние
- Проект сохранён на диск.
- Активные куски инвентаря переведены в `Reserved`.
- При загрузке — все сущности восстановлены в памяти.

### Крайние случаи
- Файл повреждён при загрузке — загружается `.backup` (если существует).
- Время сохранения: целевое < 5 сек для типичного проекта.
- Одновременное открытие проекта двумя пользователями — не поддерживается (нет конкурентного доступа); последнее сохранение перезаписывает предыдущее.

## 12. Инвентаризация куска (FurLab Access): регистрация, оцифровка, метка

**Статус:** ✅  
**Модуль:** FurLab Access

### Входное состояние
- Физический кусок меха подготовлен к оцифровке (лежит на сканере).
- На кусок нанесена (или будет нанесена) инвентарная метка `FL-SCR-XXXXXX`.

### Что происходит

**Шаг 1 — Загрузка скана (I-20):**  
Оператор загружает скан изображения куска (типично 200–300 DPI). Алгоритм сегментации мездры (Е.2) строит маску `M_mezdra`: пороговая бинаризация → морфологическое закрытие → flood fill фона → итоговая маска.

**Шаг 2 — Детекция ворса (Е.3):**  
Алгоритм ищет чёрную метку-отрезок на мездре: детектор `dark-line` (PCA по тёмным компонентам) → резервный `opencv-hough`. Конец отрезка, ближайший к краю мездры, определяет направление ворса. Результат: `napDirectionDeg` ∈ [0°, 360°).

**Шаг 3 — Оцифровка контура (I-21):**  
Из маски `M_mezdra` извлекается замкнутый полигон. Координаты переводятся в мм: `k = 25.4/DPI мм/пиксель`. Контур сохраняется в `ScrapPiece.scrapContour.path`.

**Шаг 4 — Заполнение карточки (I-22):**  
Оператор проверяет и при необходимости корректирует: `napDirectionDeg`, `scrapQuality` (`Good`/`Limited` + комментарий), `storageLocationId` (BOX-01…BOX-99), `materialId`.

**Шаг 5 — Присвоение метки и регистрация (I-01, I-23):**  
Генерируется уникальный `inventoryTag` (`FL-SCR-XXXXXX`). Создаётся запись `ScrapPiece` в БД со статусом `Available`. Транзакция `Receipt` записывается в `ScrapTransaction`. Метка (QR + штрих-код) печатается и наклеивается на кусок.

**Метрики (I-03):** автоматически вычисляются `areaMm2`, `bboxWidthMm`, `bboxHeightMm`, `maxSpanMm`.

### Выходное состояние
- Запись `ScrapPiece` в БД, `scrapStatus = Available`.
- Физический кусок помечен инвентарной меткой.
- `ScrapTransaction` типа `Receipt` с `statusAfter = Available`.

### Крайние случаи
- Скан слишком тёмный / засвеченный → алгоритм tonal-rescue применяет диапазон [118, 210] для восстановления маски. Если и это не помогает — контур вводится вручную.
- QR-наклейка на мездре попадает в зону детекции ворса — зона наклейки автоматически исключается из анализа (алгоритм Е.3.2).
- Дубликат `inventoryTag` — база отвергает вставку (UNIQUE constraint), оператору показывается ошибка.
- Кусок без детектируемого направления ворса — `napDirectionDeg = NULL`, кусок регистрируется, но будет исключён из фильтрации NFP+SA с `napToleranceDeg`.

## 13. Резервирование куска при сохранении проекта

**Статус:** ✅  
**Модуль:** FurLab Studio + FurLab Access (БД)

### Входное состояние
- Проект содержит `LayoutRunScrapPlacement` с привязанными `ScrapPiece`.
- Статус куска: `Available` (нормальный путь).

### Что происходит
1. При сохранении проекта (п.11) для каждого уникального `scrapPieceId` в `LayoutRunScrapPlacement` текущего проекта:
   a. Проверяется: существует ли уже активный `ScrapReservation` (`releasedAt = NULL`) для этого куска и этого `layoutRunId`.
   b. Если нет — создаётся запись `ScrapReservation` (`scrapPieceId`, `layoutRunId`, `fragmentId`, `reservedAt = now`, `reservedBy`).
   c. `ScrapPiece.metrics.scrapStatus` → `Reserved`.
   d. Записывается `ScrapTransaction` с `transType = Reserve`, `statusBefore = Available`, `statusAfter = Reserved`.
2. При недоступности Access БД — операция записывается в JSON-fallback; синхронизация при следующем подключении.

### Выходное состояние
- `ScrapPiece.scrapStatus = Reserved`.
- Запись `ScrapReservation` с `releasedAt = NULL`.
- `ScrapTransaction` типа `Reserve`.

### Крайние случаи
- Кусок уже `Reserved` другим проектом — транзакция `Reserve` не создаётся; пользователь получает предупреждение.
- Кусок в статусе `Used` — резервирование невозможно; проект сохраняется с предупреждением о недоступности куска.
- Отмена проекта (удаление `LayoutRun`) — активный `ScrapReservation` снимается: `releasedAt = now`, транзакция `Release`, `scrapStatus → Available`.

## 14. Lifecycle куска: Available → Reserved → Used → Discarded

**Статус:** ✅  
**Модуль:** FurLab Access (БД)

### Описание

Полный жизненный цикл единицы инвентаря `ScrapPiece` управляется через `ScrapTransaction`. Каждый переход статуса фиксируется отдельной транзакцией.

```
Available ──reserve()──► Reserved ──useConfirm()──► Used
    ▲                        │
    └──release()─────────────┘

Любой статус ──writeOff()──► Discarded
```

### Переходы

| Переход | transType | Триггер | Результат |
|---|---|---|---|
| → Available | `Receipt` | Регистрация нового куска | Кусок появляется в инвентаре |
| Available → Reserved | `Reserve` | Сохранение проекта с размещением куска | Кусок недоступен для других проектов |
| Reserved → Available | `Release` | Удаление зоны/выкладки или отмена проекта | Резерв снят, кусок снова доступен |
| Reserved → Used | `UseConfirm` | Оператор подтверждает физический раскрой | Кусок израсходован, статус окончательный |
| Любой → Discarded | `WriteOff` | Оператор регистрирует утрату/повреждение | Кусок исключён из инвентаря |
| Available → Available (смена ячейки) | `Move` | Физическое перемещение куска | Обновляется `storageLocationId` |

### Входное состояние каждой операции
- `Reserve`: `scrapStatus = Available`, существует активный `ScrapReservation` без `releasedAt`.
- `Release`: `scrapStatus = Reserved`, существует запись `ScrapReservation`.
- `UseConfirm`: `scrapStatus = Reserved` (кусок должен быть зарезервирован перед подтверждением).
- `WriteOff`: любой статус; требует `note` с причиной.
- `Move`: любой статус; указываются `fromLocId` и `toLocId`.

### Что записывается в ScrapTransaction
Каждая транзакция хранит: `transType`, `transAt`, `statusBefore`, `statusAfter`, `fromLocId`, `toLocId`, `note`, `sourceRef`.

### Крайние случаи
- Прямой переход `Available → Used` без резервирования — не предусмотрен стандартным workflow; возможен только через WriteOff + новую регистрацию или ручное исправление в Access.
- `Discarded` — терминальный статус; обратный переход не предусмотрен.
- Множественное резервирование одного куска — предотвращается проверкой активного `ScrapReservation` при сохранении проекта.
