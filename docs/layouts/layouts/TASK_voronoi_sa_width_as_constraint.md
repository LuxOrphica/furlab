## 1. Почему меняем (итог тупика PH-1b)

PH-1b застряла: exact guard роняет уже-зелёную zone_1 (R2 0.43→2.02, covF −1.6пп, 6 SAFETY-DROP). Диагноз агента верный:
- **ring-selection:** computeFragInfo берёт контур по max |area| (может взять внутреннюю дыру); Pass 3 — по max signed (outer). Guard-MBR завышен.
- **бюджетная близорукость:** каждое отдельное удаление оставляет донора ≥70, но 5–7 подряд от одного куска суммарно роняют его <70. Guard не видит суммарный бюджет.

Это **подтверждает** общий вывод, а не опровергает: **постфактум-ремонт (Expand/Absorb с guard) не способен дать ≥70.** Ширину надо вшить в само деление, а не ловить после.

### ЗАПРЕЩЕНО (агент туда уходил — больше нет)
- **Постфактум Expand/Absorb как МЕХАНИЗМ ≥70.** Снять как рабочий путь. Финальный safety-drop — только assert «не должен сработать» + лог, не нагрузка.
- **Зоно-зависимое поведение** («PH-1b только если R2>0.5» и т.п.). Overfitting/гейминг (контракт §7). Один механизм на все зоны.
- **«Честный предел» как объяснение FAIL.** По слову эксперта-скорняка: любая зона из лекал ≥70-покрываема, фита-предела нет. FAIL = дефект. Не переименовывать.
- **Искажение только по ПОКРЫТИЮ (CPT argmax) как фикс ширины.** Оно закрывает R2 (назначение клетки), но НЕ ширину территории. Ширине нужен свой якорь.

## 2. Что строим — ДВА рычага В СВЯЗКЕ

Цель: каждый фрагмент ≥70 по обеим граням MBR **по построению**; R2 → под-сетка; **absorb → 0**; covF не ниже PH-1a baseline; на ВСЕХ зонах одним механизмом.

### Рычаг A — виабельность по ширине + СВАП (слой подбора)
- Кусок **виабелен в своём месте**, если `core_k ∩ (его регион)` может держать ≥70-фрагмент. Конструктивный якорь: **эрозия региона на 35 мм непуста** (влезает диск Ø70 ⇔ MBR обе грани ≥70). Это проверка ПОСТРОЕНИЯ; судья — verify (MBR через convex hull).
- Размещённый кусок, **не виабельный там, где сидит** → **СВАП**: найти кандидата из инвентаря, чьё ядро в этом регионе виабельно (с учётом no-reuse), заменить. Источник тонких фрагментов убирается **подбором**, а не absorb'ом.

### Рычаг B — ширино-искажённая раздача (слой деления)
- Делить так, чтобы **≥70-якорь** каждого виабельного куска (его эродированный регион) **оставался за ним**. Contested-клетки отдавать кроющим, **не разрезая ни один кусок ниже его якоря**.
- Coverage-argmax (CPT) разрешает contested **только там, где не нарушает ширино-якорь**. **Якорь по ширине приоритетнее argmax по покрытию.**

Свап (A) чинит плохо-сидящие куски (источник тонких фрагментов); ширино-раздача (B) делит остальное (источник R2-gap). **Поодиночке ни один не закрывает.**

### Край и непокрытые клетки (UNCOV) — обязательно проговорить
Рычаги A/B работают с эксклюзивом и contested. **UNCOV-клетки (cov=0, ни одно ядро не кроет) — отдельная категория, A/B их не трогают.** Дизайн обязан явно сказать, что с ними:
- UNCOV остаются physMissing, классифицируются **verify по месту** в Ось2А (внутр.) / Ось2Б (край). Solver их не «присваивает» ради смены оси (это гейминг, R9 поймает).
- **Краевое кольцо шириной с припуск** ядром непокрываемо по инсету → кроется телом = Ось2Б. **covF<100 здесь ОЖИДАЕМ, не дефект** (контракт §8) — не принимать за провал и не «добивать».
- Край детали кроется **языком** заякоренного куска (INV-4): тонкий краевой язык куска с широким якорем валиден (MBR глобальный ≥70). Край не должен резаться как отдельные тонкие куски.
- **Свап и донорство не должны регрессировать Ось2Б** (судится контрактом v4.2 / verify).
