# Приложение Д. Параметры и сценарии выполнения выкладок

---

## Д.1. Типы выкладок

| Тип | Класс (`layoutType`) | Назначение |
|-----|-------|-----------|
| Регулярная | `RegularLayout` | Выкладка по паттерну в пределах контура зоны |
| Нерегулярная | `IrregularLayout` | Произвольные контуры (импорт / рисование / трафарет) |
| Заполнение остатка | `FillRemainingAreaLayout` | Алгоритмическое заполнение остаточной области зоны |
| Инвентарная (Режим А) | `InventoryLayout` | Формирование фрагментов из кусков инвентаря ScrapPiece |

> **Примечание 1.** Для `InventoryLayout` предусмотрены стратегии размещения `greedy`, `bestFit`, `manualAssist` и `nfp_sa`. Стратегия `nfp_sa` реализует автоматический подбор методом No-Fit Polygon + Simulated Annealing и хранится в параметре `placementStrategy`.

> **Примечание 2.** Режим Б («назначение кусков», `ScrapAssignment`) рассматривается как операционный сценарий постобработки, а не как значение поля `layoutType`. В этом режиме `ScrapPiece` назначаются на уже сформированные `Fragment` с фиксацией связи через `LayoutRunScrapPlacement`. Параметры режима Б описаны в Д.6.

## Д.5. Параметры InventoryLayout (Режим А)

| Параметр | Тип | Описание |
|----------|-----|---------|
| `placementStrategy` | enum | `greedy` / `bestFit` / `manualAssist` / `nfp_sa` |
| `normalizeRules` | JSON | Правила нормализации |

Дополнительная конфигурация передаётся через `InventoryLayoutConfig`:

| Параметр | Тип | Описание |
|----------|-----|---------|
| `maxCandidates` | int | Максимальный размер пула ScrapPiece на прогон |
| `filters` | JSON | Фильтры отбора кусков: материал, направление ворса, площадь, качество |
| `constraints` | JSON | Геометрические ограничения: допустимые повороты, допуски, припуски на швы |

---

## Д.5а. Параметры стратегии nfp_sa для InventoryLayout

При `placementStrategy=nfp_sa` в `InventoryLayoutConfig.constraints` передаются дополнительные параметры алгоритмического решателя:

| Параметр | Тип | Описание |
|----------|-----|---------|
| `maxSolveMs` | number | Максимальное время решения SA-цикла, мс (дефолт: 90 000) |
| `seed` | number, NULL | Зерно генератора случайных чисел для воспроизводимости |
| `seamAllowanceReserveMm` | number | Производственный припуск на шов, мм; используется при построении ядра куска (inset). Производственный стандарт: 3 мм |
| `napTargetDeg` | number | Целевое направление ворса зоны, градусы |
| `napToleranceDeg` | number | Допуск направления ворса, градусы (дефолт: 15) |
| `minWidthMm` | number | Минимальная ширина ядра куска после припуска, мм (дефолт: 0) |
| `minLengthMm` | number | Минимальная длина ядра куска после припуска, мм (дефолт: 0) |
| `maxCandidates` | int | Общий параметр `InventoryLayoutConfig`; для стратегии `nfp_sa` рекомендуемое значение по умолчанию — 300 |

Пул кандидатов передаётся в поле `inputs.candidates` — массив объектов `{ scrapPieceId, scrapContour, napDirection (napDirectionDeg в физической таблице БД), quantity }`.

## Д.6. Параметры ScrapAssignment (Режим Б)

В режиме Б новая геометрия фрагментов не формируется; выполняется назначение инвентарных кусков на уже существующие `Fragment`. Конфигурация:

| Параметр | Тип | Описание |
|----------|-----|---------|
| `maxCandidates` | int | Максимальный размер пула ScrapPiece |
| `filters` | JSON | Фильтры (качество, место хранения, материал) |
| `constraints` | JSON | Ограничения на поворот, смещение и совместимость размещения |

---

## Д.7. NormalizeRules — общий блок для всех типов

| Параметр | Тип | Описание |
|----------|-----|---------|
| `minFragmentWidthMm` | number, NULL | Минимальная ширина фрагмента, мм |
| `minFragmentLengthMm` | number, NULL | Минимальная длина фрагмента, мм |
| `simplifyToleranceMm` | number, NULL | Допуск упрощения контура, мм |
| `mergeSmallFragments` | boolean, NULL | Объединять мелкие фрагменты |
| `seamAllowanceReserveMm` | number, NULL | Резерв на припуск шва, мм |

## Д.8. Сценарии выполнения выкладок

### Сценарий 1. Регулярная выкладка по паттерну

**Тип:** `RegularLayout`

**Входные данные:** зона с заданным контуром; паттерн «ёлочка»; параметры паттерна (шаг 45 мм, угол 30°); `normalizeRules` — `minFragmentWidthMm=20`, `seamAllowanceReserveMm=5`.

**Процесс:**
1. Генерация структуры паттерна в пространстве зоны.
2. Отсечение результата по `zoneContour`.
3. Применение `normalizeRules`: упрощение контуров, удаление фрагментов шириной менее 20 мм.
4. Формирование набора фрагментов с атрибутами `fragmentContour` и метриками площади.

**Результат:** набор нормализованных фрагментов, готовых к спецификации раскроя.

**Фиксация данных:** `Layout.params` → `LayoutRun.paramsSnapshot` → `Fragment.fragmentContour`.

---

### Сценарий 2. Нерегулярная выкладка по трафарету

**Тип:** `IrregularLayout`, `sourceType=stencil`

**Входные данные:** контур зоны; `stencilId` — ссылка на шаблон кроя; `simplifyToleranceMm=2.5`; `normalizeRules` с `mergeSmallFragments=true`.

**Процесс:**
1. Загрузка трафарета по `stencilId`.
2. Упрощение контуров с допуском 2.5 мм.
3. Валидация замкнутости контуров.
4. Отсечение по контуру зоны.
5. Объединение фрагментов менее `minFragmentWidthMm`.

**Результат:** набор фрагментов со сглаженными контурами, соответствующих форме трафарета.

**Фиксация данных:** `Layout.params` (`sourceType`, `stencilId`, `simplifyToleranceMm`) → `LayoutRun.paramsSnapshot` → `Fragment.fragmentContour`.

---

### Сценарий 3. Заполнение остатка методом Вороного

**Тип:** `FillRemainingAreaLayout`, `algorithm=Voronoi`

**Входные данные:** остаточная область зоны; `seedCount=8`, `relaxIters=3`, `minCellAreaMm2=500`; `normalizeRules`.

**Процесс:**
1. Размещение зёрен в зоне.
2. Построение диаграммы Вороного.
3. Итеративная релаксация (3 итерации) для выравнивания ячеек.
4. Отброс ячеек с площадью < 500 мм².
5. Отсечение по контуру зоны.
6. Применение `normalizeRules`.

**Результат:** набор фрагментов, равномерно покрывающих остаточную площадь зоны.

**Фиксация данных:** `Layout.params` (`algorithm=Voronoi`, `algorithmParams`) → `LayoutRun.paramsSnapshot` → `Fragment.fragmentContour`.

---

### Сценарий 4. Инвентарная выкладка (Режим А)

**Тип:** `InventoryLayout`

**Входные данные:**
- Контур зоны.
- `maxCandidates=50`, `filters` — совпадение `materialId`, направление ворса в пределах ±15°, `scrapStatus=Available`.
- `constraints` — допустимые повороты 0° и 180°, `maxOffsetMm=10`, `allowMirror=false`, `allowFlip=false`.
- `placementStrategy=bestFit`.
- `normalizeRules` — `minFragmentWidthMm=15`, `mergeSmallFragments=true`.

**Процесс:**
1. Отбор пула ScrapPiece по фильтрам (не более 50 кандидатов).
2. Проверка совместимости геометрии кусков с ограничениями поворота и смещения.
3. Выполнение алгоритма `bestFit`: подбор куска с наилучшим перекрытием незанятой области зоны.
4. Отсечение контуров фрагментов по `zoneContour`.
5. Нормализация: объединение мелких фрагментов.

**Результат:** набор фрагментов, производных от конкретных кусков инвентаря; полная трассируемость «фрагмент → физический кусок».

**Фиксация данных:**
- `Layout.params` (`placementStrategy`, `normalizeRules`), `InventoryLayoutConfig` (`maxCandidates`, `filters`, `constraints`).
- `LayoutRun.paramsSnapshot` — полный снимок параметров для воспроизводимости.
- `LayoutRunScrapPlacement` — записи (`fragmentId`, `scrapPieceId`, `rotationDeg`, `offsetXmm`, `offsetYmm`, `resultContourSnapshot`).

---

### Сценарий 5. Ручное назначение кусков к фрагментам (Режим Б)

**Тип:** операционный сценарий постобработки (не `layoutType`); выполняется поверх уже сформированных `Fragment`.

**Предусловие:** фрагменты уже существуют (результат любой предшествующей выкладки). Для фиксации назначения создаётся отдельный `LayoutRun` назначения либо используется `LayoutRun` исходной выкладки.

**Входные данные:** сформированные фрагменты из предыдущей выкладки; `maxCandidates=30`; `filters` — `scrapQuality=Good`, `StorageLocation.locCode=BOX-01`; `constraints` — только поворот 0°.

**Процесс:**
1. Отбор доступных кусков по качеству, месту хранения и материалу.
2. Жадное сопоставление: назначение каждому фрагменту наилучшего доступного куска.
3. Фиксация трансформации (поворот, смещение).
4. Сохранение `resultContourSnapshot` после отсечения.

**Результат:** записи `LayoutRunScrapPlacement`, связывающие фрагменты с физическими единицами инвентаря; изменение статуса ScrapPiece `Available → Used`.

**Фиксация данных:**
- `LayoutRunScrapPlacement` (`fragmentId`, `scrapPieceId`, `resultContourSnapshot`).
- `ScrapTransaction` — смена статуса.
- `ScrapReservation` — подтверждение или снятие резерва.

---

### Сценарий 6. Автоматический подбор кусков (NFP+SA)

**Тип:** `InventoryLayout`
**Стратегия размещения:** `placementStrategy=nfp_sa`

**Входные данные:**
- Контур зоны.
- Пул кандидатов: до 300 кусков инвентаря с контурами и направлениями ворса.
- `seamAllowanceReserveMm=3` (производственный припуск), `napTargetDeg` — целевое направление ворса, `napToleranceDeg=15`.
- `maxSolveMs=90000`.

**Процесс:**
1. Растеризация зоны и каждого кандидата в бинарные маски с шагом ячейки 3 мм. NFP аппроксимируется через маски занятости: допустимые позиции куска определяются по отношению к `zoneMask`, а пересечение с `occupiedMask` используется для расчёта штрафа за перекрытие и отбора предпочтительных размещений.
2. Построение начального жадного решения (Greedy Warm Start): сортировка кандидатов по площади, 16 попыток случайного размещения каждого, формирование начального решения.
3. Имитация отжига (Simulated Annealing, SA): оптимизация размещений с использованием операторов изменения решения: TRANSLATE, ROTATE, SWAP, ADD с критерием принятия Метрополиса.
4. Постобработка: для каждого куска вычисляется ядро (`T_i(inset(p_i, seamAllowanceReserveMm)) ∩ Z`), вычитается уже занятое пространство; если результат вычитания состоит из нескольких связных компонент, каждая компонента оформляется как отдельный фрагмент; куски с пустым результатом исключаются из набора.
5. Формирование полного набора фрагментов с контурами `inZoneCoreContour` (ядро) и `alignedContour` (по припуску).

**Результат:** набор фрагментов с трассируемостью «фрагмент → ScrapPiece»; коэффициент покрытия ядрами `coveredRatio`.

**Фиксация данных:** применение (`Apply`) фиксирует размещения в `LayoutRunScrapPlacement`; до подтверждения операции результат имеет статус предварительного решения и не фиксируется в постоянном хранилище.

## Д.9. Запись трассируемости размещения

Во всех сценариях, предполагающих применение инвентарного куска (`InventoryLayout`, включая стратегию `nfp_sa`, и `ScrapAssignment`), факт использования `ScrapPiece` фиксируется записью в таблице `LayoutRunScrapPlacement`:

```
{
  id:                      UUID,   // Идентификатор записи размещения
  layoutRunId:             UUID,   // Идентификатор прогона выкладки
  fragmentId:              UUID,   // Идентификатор фрагмента
  scrapPieceId:            UUID,   // Идентификатор куска инвентаря
  rotationDeg:             number, // Угол поворота при размещении
  offsetXmm:               number, // Смещение по X, мм
  offsetYmm:               number, // Смещение по Y, мм
  resultContourSnapshot:   JSON    // Итоговая геометрия после отсечения
}
```

**Гарантия воспроизводимости:** `LayoutRun.paramsSnapshot` фиксирует все входные параметры в момент выполнения. Совместно с фиксированным `layoutRunId` это обеспечивает детерминированное повторное выполнение.
