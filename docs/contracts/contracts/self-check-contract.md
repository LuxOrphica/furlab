# Контракт самопроверки и проверки последствий кода

Обязательный регламент для исполнителя.

## Общий принцип

Любое изменение кода считается выполненным только после проверки его последствий.  
Исполнитель обязан проверять не только исправляемый участок, но и все элементы системы, на которые изменение могло повлиять прямо или косвенно.

Передача результата без проверки последствий считается незавершенной задачей.

## Обязанности исполнителя

Перед демонстрацией результата исполнитель обязан:

- определить, какие файлы и участки кода были изменены;
- определить, какие функции, экраны, разделы или процессы могли быть затронуты;
- самостоятельно проверить все связанные сценарии;
- убедиться, что изменения не вызвали регрессии.

Исполнитель не имеет права передавать результат в формате «вот код, проверьте».  
Исполнитель обязан передать результат вместе с отчетом о проверке.

## Обязательная самопроверка перед сдачей

Перед показом результата исполнитель обязан проверить:

### Функциональность

- основной сценарий, ради которого вносилось изменение;
- обратный сценарий (возврат к предыдущему состоянию);
- граничные состояния (пустые значения, крайние параметры, ошибки).

### Интерфейс (если изменение касается UI)

- соседние элементы интерфейса;
- выравнивание и отступы;
- `hover` / `active` / `disabled` состояния;
- отображение скрываемых и показываемых блоков;
- адаптивность.

### Системные проверки

- отсутствие ошибок в консоли;
- корректность кодировки текстов;
- корректность загрузки данных;
- корректность работы связанных функций.

### Интеграционные проверки

- сохранение данных;
- повторное открытие страницы;
- обновление страницы;
- повторное выполнение действия.

## Проверка последствий изменения

Исполнитель обязан анализировать зоны риска и проверять:

- связанные функции;
- соседние кнопки и элементы;
- формы ввода;
- отображение разделов;
- обработку ошибок;
- работу после перезапуска или обновления.

Исправление одной точки без проверки связанных элементов запрещено.

## Обязательный отчет по задаче

Каждая задача передается заказчику только вместе со следующим отчетом:

**Задача:**  
**Что изменено:**  
**Измененные файлы:**  
**Какие элементы могли быть затронуты:**

**Проверено:**

- основной сценарий: `OK / BUG`
- связанные элементы: `OK / BUG`
- интерфейс и верстка: `OK / BUG`
- видимость разделов: `OK / BUG`
- консоль без ошибок: `OK / BUG`
- кодировка текстов: `OK / BUG`
- сохранение данных: `OK / BUG`
- повторное выполнение сценария: `OK / BUG`

**Что не проверено:**

**Потенциальные риски:**

**Доказательства проверки:**  
(скриншоты / видео / логи)

Задача без такого отчета считается не переданной.

## Правило регрессий

Если после сдачи задачи обнаруживается проблема в очевидно связанном участке системы, который должен был быть проверен при самопроверке, такая проблема считается регрессией.

Исполнитель обязан исправить регрессию в рамках той же задачи без дополнительной оплаты.

## Критерий завершения задачи

Задача считается завершенной только при выполнении всех условий:

- изменение реализовано;
- проведена самопроверка;
- проверены последствия изменения;
- предоставлен отчет проверки;
- подтверждено отсутствие регрессий.

До выполнения этих условий задача считается находящейся в работе.

## Базовый принцип работы

Исполнитель обязан исходить из следующего принципа:

Любое изменение кода может повлиять на систему шире, чем предполагается.  
Поэтому каждая правка должна сопровождаться проверкой ее последствий.

Исправление без проверки последствий считается технически неполной работой.
