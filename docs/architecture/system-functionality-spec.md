# FURLAB AC — текущий функционал (срез на 2026-05-23)

Документ фиксирует фактически реализованный функционал системы FURLAB AC на текущий момент в репозитории `furlab-access`.

## 1. Назначение системы

FURLAB AC решает задачи:
- загрузка сканов лоскутов;
- автоматическое/ручное определение контура и направления ворса;
- ручная доработка контура;
- запись карточки лоскута в Access;
- ведение реестра и карточек лоскутов;
- статусы жизненного цикла (доступен/резерв/использован/списан);
- история размещений;
- отчеты (сводка, ведомость отбора, трассируемость);
- накопление обучающего датасета контуров.

## 2. Архитектура (коротко)

- Frontend: React + Ant Design (SPA).
- Legacy-движок внутри экрана сканов: `src/legacy/*` (canvas, сегментация, редактор контура).
- Backend: Node.js HTTP API на `5500` (`tools/ui_lab_server.js`).
- База: MS Access (`BD/Furlab 1.accdb`).
- Интеграция с Access: через скрипты `scripts/access_*` и cscript/DAO/ADO.
- Отдельное хранилище обучающих аннотаций: `data/training/annotations.ndjson`.

## 3. Навигация в UI (основные разделы)

В приложении есть 5 функциональных блоков:
- `Загрузка сканов`;
- `Инвентарь лоскутов` (реестр + карточка);
- `История размещений`;
- `Отчёты` (Сводка / Ведомость отбора / Трассируемость);
- `QR-генератор` (генерация и печать QR-меток для лоскутов).

Текущая версия фронта:
- `2.0.42`.

## 4. Экран «Загрузка сканов»

### 4.1 Загрузка и режимы сканирования

Поддерживаются две кнопки загрузки:
- `Прототип` (legacy baseline);
- `Скан` (текущий рабочий pipeline V3).

После выбора файла:
- файл отображается в рабочем поле;
- запускается обработка контура/направления;
- показывается inline-статус обработки (pending/success/error);
- телеметрия выводится в верхней строке и в «Тех информации».

### 4.2 Определение направления ворса

Режимы:
- `Авто`;
- `Ручной выбор`.

Доступна кнопка `Очистить` для сброса точки/линии направления.

### 4.3 Отображение в рабочем поле

Чекбоксы отображения (справа в области превью):
- `Сетка`;
- `Контур`;
- `Габариты`;
- `Точки`;
- `Стрелка`;
- `Маска`.

Поддерживаются:
- вертикальный zoom-slider;
- колесо мыши для масштабирования;
- центрирование/панорамирование сцены.

### 4.4 Редактор контура (ручная правка)

Реализован интерактивный редактор с режимом редактирования:
- включение/выключение `Ред. контур`;
- сброс черновика `Сброс контура`;
- применение черновика `Применить контур`;
- выбор/удаление точек;
- массовое выделение рамкой.

Важно:
- расчет геометрии (площадь/габариты/макс. габарит) берется из примененного контура;
- при не-примененном черновике в запись уходит предыдущее примененное состояние.

### 4.5 Параметры куска и валидация

Поля карточки на экране сканов:
- инв. метка (`FL-SCR-XXXXXX`);
- площадь, ширина/высота, макс. габарит;
- угол ворса;
- материал;
- локация;
- качество;
- комментарий.

Валидации:
- формат инвентарной метки обязателен;
- обязательные поля для записи в Access;
- для ограниченного качества обязателен комментарий.

### 4.6 Запись в Access

Кнопка `Записать в Access`:
- создает новую запись либо обновляет существующую (по `inventoryTag`);
- сохраняет контур, метрики, статус/качество, материал/локацию и note;
- поддерживает контроль конфликтов статусов;
- есть защита от дублей и сценарий `already_exists`.

### 4.7 Обучающий режим и датасет

Есть переключатель `Обучающий режим`.

В обучающем режиме доступна запись в обучающий датасет:
- кнопка `В обучающий датасет`;
- запись в `annotations.ndjson`;
- проверка валидности контура;
- защита от повторной записи одного и того же контура (по подписи image+contour hash);
- поддержка `overwrite`.

Что сохраняется в датасет:
- `inventoryTag`, `sourceImageName`, размеры изображения;
- `contourPoints`;
- версия пайплайна/алгоритма;
- note;
- metrics.

## 5. Раздел «Инвентарь лоскутов»

### 5.1 Реестр

Функции реестра:
- серверная пагинация;
- поиск;
- фильтры по качеству/статусу/материалу/локации;
- сортировка;
- кеширование и refresh;
- настраиваемые видимые колонки;
- сохранение фильтров в `sessionStorage`;
- быстрый переход в карточку лоскута.

Колонки включают:
- инв. метку;
- материал;
- макс. габарит;
- площадь;
- качество;
- статус;
- локацию;
- дату обновления;
- комментарий;
- действия.

### 5.2 Карточка лоскута

Карточка поддерживает:
- редактирование параметров (материал, локация, качество, комментарий);
- смену статуса через бизнес-переходы (`reserve`, `release`, `use`);
- просмотр геометрии и превью контура;
- просмотр резерва/истории по куску;
- копирование диагностической информации;
- режим нормализации отображения.

Нормализация контура в карточке:
- есть представление `Скан мездры` и `Контур выкладки`;
- применяется поворот по направлению ворса;
- для `scanSide=leather_up` применяется зеркалирование для канонического вида выкладки;
- канонизированный контур и угол ворса отдаются через read-service и используются в превью.

## 6. Раздел «История размещений»

Функционал:
- загрузка общего журнала использования;
- поиск по инв. метке/запуску/фрагменту;
- табличный просмотр:
  - дата/время;
  - запуск выкладки;
  - фрагмент;
  - поворот;
  - смещения X/Y;
  - снимок контура результата.

Трассируемость размещений логируется в `data/placements/` в формате NDJSON для аудита.

## 7. Раздел «Отчеты»

Режимы отчетов:
- `Сводка`;
- `Ведомость отбора`;
- `Трассируемость`.

Реализовано:
- агрегации по статусам/качеству/материалам/локациям;
- статистические карточки (кол-во, площадь и др.);
- табличные отчеты;
- экспорт PDF (для ведомости и трассируемости).

## 8. Backend API (фактически доступные маршруты)

Системные:
- `GET /api/health`
- `GET /api/dicts?refresh=1`
- `GET /api/export/latest-zip` — proxy к furlab-web-plugin, отдаёт последний ZIP экспорта

Реестр:
- `GET /api/registry`

История:
- `GET /api/history/usage`

Карточка/кусок:
- `GET /api/piece-exists?inventoryTag=...`
- `GET /api/piece/:id`
- `GET /api/piece/:id/reservation`
- `GET /api/piece/:id/history`
- `GET /api/piece/:id/contour`
- `POST /api/piece/:id/reserve`
- `POST /api/piece/:id/release`
- `POST /api/piece/:id/use`
- `POST /api/piece/:id/update`
- `POST /api/save-scrap-piece`

Обучающий датасет:
- `GET /api/training/annotations`
- `POST /api/training/annotations`
- `GET /api/training/stats`

## 9. Кеширование и производительность

Реализованы уровни кеша:
- клиентский cache в экранах (реестр/контуры);
- серверный memory cache;
- серверный disk cache для некоторых read-path;
- fallback на stale-cache при сбоях чтения Access.

Есть телеметрия времени:
- сеть/json/script/backend;
- диагностические поля `diag` в ответах.

## 10. Бизнес-правила статусов

Статусы поддерживаются:
- `Available`;
- `Reserved`;
- `Used`;
- `Discarded`.

Переходы проходят через отдельные write-операции с проверками.

При записи существующего куска учитывается текущий статус и ограничения overwrite (через `status_rules`).

## 11. Работа с кодировкой и локализацией

UI русифицирован.

В проекте есть служебные проверки/скрипты для контроля mojibake и UTF-8 (`tools/check_mojibake.ps1`), но при некорректной перекодировке источников возможны артефакты текста.

## 12. Дистрибуция и запуск

Поддерживаются варианты запуска:
- обычный локальный сервер (`tools/start_ui_lab_server.ps1`);
- portable EXE launcher;
- Electron portable shell;
- **one-command demo setup**: `INSTALL.ps1` + `START_DEMO.ps1` — разворачивает и запускает всё окружение одной командой.

Фронт (`line-direction-visualizer-react`) работает как SPA, поэтому разные разделы открываются по одному базовому URL, а переключение выполняется роутингом на клиенте.

Канонический base path фронта: `/furlab-ac` (legacy URLs совместимы через redirect).

## 13. Тесты и контроль качества

В репозитории есть:
- API-контрактные тесты (`tools/tests/api_contract.test.js`);
