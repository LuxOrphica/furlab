# WORKFLOW: React vs Access

Документ фиксирует разделение ответственности между React-разработкой и Access/backend.

## Для React-разработчика (клиент)

1. Не открывать `.accdb` в Access.
2. Не запускать `cscript`-скрипты к базе вручную.
3. Работать только через проектный API-маршрут `/api/*` активного backend-сервера.
4. Запускать только через согласованные project tasks в VS Code:
   - `LDV React: start all`
5. Не запускать отдельно `node tools/ui_lab_server.js`, если task уже используется.
6. Перед стартом проверять, что нет старого API-процесса на рабочем порту backend-сервера:
   - проверить занятость рабочего порта backend-сервера
   - если порт занят, остановить старый `node` по PID.
7. Рабочие порты задаются конфигурацией проекта:
   - API: проектный backend-порт
   - React: `5173`
8. Не менять `FURLAB_DB_PATH` и не переназначать путь к базе без согласованной необходимости.
9. Рабочая БД: `Furlab 1.accdb`.
10. При ошибках `dicts_not_ok`, кракозябрах, lock-проблемах:
    - не чинить БД самостоятельно,
    - сразу передать backend/Access владельцу.

## Зона ответственности React

1. UI/React-компоненты.
2. Верстка и стили.
3. Клиентская логика и интеграция с API.

## Что не входит в React-зону

1. Access-модули.
2. Таблицы и запросы Access.
3. VBA-код.
4. Структура и ручная правка файлов базы.

## Для backend/Access-владельца

1. Поддерживает совместимость API с фронтом.
2. Управляет состоянием базы и доступом к ней.
3. Отвечает за исправления в Access/VBA/скриптах работы с БД.
