# PRD — Программный комплекс FurLab

**Версия:** 1.0  
**Дата:** 2026-06-01  
**Статус:** Черновик  

---

## 1. Назначение и контекст

### 1.1 Проблема

Проектирование меховой одежды остаётся преимущественно ручным процессом. Конструктор-технолог работает с физическими шкурками, подбирая их вручную по направлению ворса, размеру и качеству. Меховые отходы (обрезки, выпады, остатки) не учитываются цифровым образом и как правило выбрасываются, несмотря на высокую стоимость материала. Отсутствует воспроизводимость проектного решения: результат зависит от опыта мастера и не фиксируется в цифровой форме.

Существующие САПР одежды (CLO 3D, Optitex, Style3D) решают задачи симуляции посадки и визуализации, но не обеспечивают:
- формализацию участков детали как зон меховой сборки;
- параметрическое формирование фрагментов внутри зоны;
- учёт направления ворса при расчёте выкладки;
- цифровой учёт инвентаря меховых кусков и отходов;
- трассируемость от физического куска к проектному фрагменту.

### 1.2 Решение

FurLab — программный комплекс цифрового сопровождения проектирования меховой одежды, состоящий из трёх взаимосвязанных инструментов:

1. **FurLab Studio** — веб-плагин, встраиваемый в рабочую среду CLO 3D. Обеспечивает зонирование деталей, параметрическую генерацию меховых фрагментов, выкладку из инвентаря и подготовку выходных данных.

2. **FurLab Access (БД)** — база данных инвентаря меховых кусков на платформе Microsoft Access. Обеспечивает цифровой учёт, хранение, резервирование и трассируемость физических кусков меха.

3. **FurLab Access** — приложение Microsoft Access с VBA-интерфейсом для инвентаризации меховых кусков. Включает формы загрузки и обработки сканов, карточки кусков, реестр инвентаря, отчётность и анализ трассируемости. Использует ту же БД что и FurLab Studio.

### 1.3 Целевая аудитория

| Роль | Инструмент | Основные задачи |
|---|---|---|
| Конструктор-технолог меховой одежды | FurLab Studio | Зонирование лекал, генерация фрагментов, выкладка, спецификация |
| Дизайнер | FurLab Studio | Визуализация меховой поверхности, подбор материала, композиция |
| Технолог / зав. производством | FurLab Access (БД) | Учёт инвентаря кусков, резервирование, трассируемость, отчётность |
| Оператор инвентаризации | FurLab Access | Оцифровка сканов, заполнение карточек кусков, печать меток |

**Базовые предположения о пользователе:**
- Регулярно работает в CLO 3D
- Знаком с понятиями: зона, деталь, лекало, ворс, выкладка
- Не является программистом

### 1.4 Ограничения текущей версии (прототип)

- Реализован как автономный веб-сервер (Node.js, порт 5600), а не как нативный плагин CLO 3D
- Работает только на Windows (зависимость от Microsoft Access и CScript/VBScript)
- Интеграция с CLO 3D через обмен файлами (DXF импорт / DXF+GLTF экспорт), не через прямой API
- Access БД требует Microsoft Access 2016+

## 2. Пользователи и сценарии

### 2.1 Персонажи

**Анна, конструктор-технолог**  
Работает в ателье или небольшом меховом производстве. Проектирует изделия в CLO 3D, выполняет выкладку вручную. Тратит много времени на подбор шкурок под зону, контроль направления ворса. Хочет видеть расчётное покрытие зоны до раскроя.

**Дизайнер Карима**  
Разрабатывает коллекции, работает с дизайн-образами в CLO 3D. Нужна быстрая визуализация различных типов меховой поверхности на изделии — радиальная, ёлочка, интарсия. Точные метрики менее важны чем скорость итерации.

**Ильяс, технолог производства**  
Отвечает за учёт материальных остатков. Сейчас ведёт учёт в Excel или на бумаге. Куски хранятся без цифровых меток, подбираются вручную. Хочет знать какие куски зарезервированы под проект, какие свободны.

### 2.2 Ключевые сценарии использования

**Сц-1: Зонирование и генерация фрагментов (регулярная выкладка)**  
Конструктор импортирует DXF лекала → размечает зоны → назначает материал и тип выкладки → получает фрагменты с метриками → экспортирует спецификацию.

**Сц-2: Выкладка из инвентаря (автоматическая, NFP+SA)**  
Конструктор выбирает зону → запускает алгоритм подбора из инвентаря → солвер подбирает куски с учётом направления ворса и геометрии → пользователь подтверждает → куски резервируются в БД.

**Сц-3: Ручная выкладка из инвентаря**  
Конструктор вручную выбирает куски из инвентаря и размещает их в зоне → фиксирует результат → получает спецификацию с трассируемостью.

**Сц-4: Инвентаризация меховых кусков**  
Технолог сканирует/вводит куски → оцифровывает контуры → присваивает метки (FL-SCR-XXXXXX) → куски доступны для подбора в Plugin.

**Сц-5: Экспорт в CLO 3D**  
Конструктор экспортирует результат выкладки как DXF лекала для CLO 3D → продолжает работу в CLO.

## 4. Модель данных (краткая)

### 4.1 Основные сущности

```
Zone          — область на детали; имеет materialId, napDirectionDeg
Layout        — конфигурация выкладки для зоны; layoutType + params JSON
LayoutRun     — зафиксированный запуск выкладки; paramsSnapshot
Fragment      — геометрический результат внутри зоны; fragmentContour, areaMm2
FurMaterial   — паспорт мехового материала
ScrapPiece    — инвентарный кусок; inventoryTag, scrapContour, scrapStatus
LayoutRunScrapPlacement — факт применения куска: layoutRunId + fragmentId + scrapPieceId
ScrapReservation        — активное резервирование куска под проект
ScrapTransaction        — полный журнал аудита операций с куском
StorageLocation         — ячейка физического хранения
Specification / SpecItem — отчётный слой агрегации фрагментов
```

### 4.2 Типы выкладки (layoutType)

| Тип | Описание | Статус |
|---|---|---|
| `RegularLayout` | Регулярная (продольная, радиальная, ёлочка, смещение) | ✅ |
| `IrregularLayout` | Интарсия, импорт контуров, мозаика Вороного | ✅ |
| `InventoryLayout` | Из инвентаря (ручная, NFP+SA) | ✅ |

### 4.3 Статусы инвентарного куска

```
Available ──reserve()──► Reserved ──use()──► Used
    ▲                        │
    └──release()─────────────┘
    
Любой статус ──writeOff()──► Discarded
```

Подробный состав атрибутов — см. **Приложение Г** (ScrapPiece schema) и **Приложение В** (Data Model Glossary).

## 5. Нефункциональные требования

### 5.1 Производительность

| Требование | Целевое значение |
|---|---|
| Время генерации регулярной выкладки (зона ~370к мм²) | < 2 сек |
| Время работы NFP+SA солвера (300 кандидатов) | < 60 сек |
| Время загрузки инвентаря (до 1000 кусков) | < 3 сек |
| Время сохранения проекта | < 5 сек |

### 5.2 Точность расчётов

- Площадь фрагментов: погрешность ≤ 0.1% от расчётной
- Покрытие зоны: метрика coveredRatio точна до 4 знаков
- Направление ворса: допуск задаётся пользователем (по умолчанию ±15°)

### 5.3 Платформа и среда

- ОС: Windows 10/11 (обязательно для Access + CScript)
- Браузер: Chrome/Edge последние версии
- CLO 3D: 2024.x и выше
- Microsoft Access: 2016+
- Node.js: 18+

### 5.4 Надёжность

- Проект сохраняется с резервной копией (`.backup`)
- При недоступности Access БД — fallback на JSON-файлы для резерваций
- Солвер не блокирует UI (SSE-поток прогресса)

### 5.5 Расширяемость

- Новый тип выкладки добавляется как отдельный модуль в `src/modes/`
- Библиотека материалов расширяется без изменения архитектуры
